YESAB Map Follow-Up - 2026-05-05

What Changed Since The April Journal Entry

The April 15 report captured the first working map/API hybrid. Since then, the project has moved from enrichment-only map output to a broader reproducible artifact set: the API cache is compressed and refreshed, API-only projects can be shown as approximate points, the same joined payload can be exported as a GeoPackage, and the map-building code now shares core behavior through a tested package.

The practical shift is that most cached registry records now appear somewhere in the output: 4,497 of 4,617 cached API projects are mapped either through shapefile geometry or API-derived approximate points.

1. Cache Format And Refreshes

The API cache moved from raw JSON files to Zstandard-compressed JSON. Bucket metadata in data/api/state.json now records compressed content type, content length, hashes, fetch times, and record counts.

  • The merged API dataset is now projects_merged.json.zst.
  • Seven buckets are tracked, including focused 2025 and 2026 refresh buckets.
  • The current merged cache was regenerated at 2026-05-05T19:08:51Z.

2. API-Only Fallback Points

Projects that do not match shapefile geometry can now be mapped from API location coordinates when a usable coordinate is available. These are kept distinct from shapefile-backed features.

  • A new API_Approximate_Points layer is emitted by both map builders.
  • Details panels label these records as approximate API locations.
  • QA reports now split projects into matched, fallback, and unmapped groups.

3. Coordinate Classification

Fallback coordinates are classified so downstream users can distinguish plausible API coordinates from generic or suspicious values.

  • Coordinate classes and flags are included in fallback feature properties and QA summaries.
  • data/api/location_overrides.csv supports explicit coordinate replacement for known cases.
  • One current override corrects project 2024-0009.

New Export Path

The joined map payload can now be written as an enriched GeoPackage through scripts/build_geopackage.py. This gives the project a GIS-friendly output in addition to the browser map artifacts.

Output Location What It Carries
GeoPackage out/yesab-projects.gpkg Enriched feature tables, API join status, approximate API point source labels, and non-spatial QA/source tables.
Single-file map out/yesab-map-in-one.html Self-contained browser output with shapefile features, matched API metadata, and API fallback points.
Split map out/yesab-map/ Multi-file static site with separate layer data files, including API_Approximate_Points.js.

Shared Code And Tests

The duplicated builder logic was pulled into yesab_map/core.py, with both map builders and the GeoPackage exporter using the same helper path for joins, labels, cache loading, fallback point creation, coordinate overrides, and QA summaries.

  • Scripts now live under scripts/.
  • Shared behavior lives under yesab_map/.
  • Regression tests live in tests/test_yesab_map_core.py.
  • Repo instructions now explicitly call for red-green TDD and shared-helper coverage for join, QA, fallback, and details-panel behavior.

Share Package

A share landing page was added at out/index.html. It gives people a simpler entry point for the browser artifacts instead of requiring them to know the exact output filenames.

  • Static previews were added for both single-file and split map builds.
  • Active-state previews show the map with project details open.
  • A short browser demo video was added under out/previews/.
  • Launch-link spacing was adjusted after the initial page landed.
