YESAB Map + API Follow-Up · 2026-04-15

What Has Been Built Since The First Report

The first report framed options for tying the YESAB registry API into the static project-map viewer. Since then, the project has moved from exploration to implementation: a local API cache exists, both map builds consume it, QA artifacts are generated, registry-linked features are highlighted in the map, and the UI now exposes source metadata and direct registry links.

The practical result is a hybrid model: the shapefile archive remains the geometry backbone, while the YESAB registry API now provides live-ish status and metadata enrichment through a local cache.

1. API Cache Layer

A lightweight bucketed cache was added in cache_yesab_api.py. It stores year-window API responses under data/api/buckets/, tracks state in data/api/state.json, and emits a merged dataset at data/api/projects_merged.json.

  • Designed to avoid full refetches.
  • Works by year buckets rather than a complex per-project sync engine.
  • Known constraint: refresh must be sequential because runs share one state file.

2. Map Enrichment

Both builders now load the merged API cache when present and join registry records to shapefile features by project-number style fields already present in the map data.

  • Registry-linked features are visually highlighted in gold.
  • Details panels show stage, outcomes, proponent, districts, sectors, and more.
  • Details now link to both the API endpoint and the human-facing registry page.

3. QA Layer

QA outputs were added to make the shapefile/API overlap inspectable instead of assumed. This is important because the two sources do not line up perfectly.

  • Matched and unmatched API projects are summarized in JSON and HTML.
  • Per-project layer coverage is reported for matched records.
  • The mismatch list helps distinguish real gaps from join-field issues.

UX And Source Clarity

After enrichment was in place, the map UI received targeted fixes and traceability improvements.

  • Hover popup jitter in narrow windows was fixed by constraining tooltip placement to the map viewport.
  • The popup now tracks canvas hover rather than global mouse movement.
  • An About data sources panel was added to both builds.
  • The About panel now shows page build date, map file date, and registry cache date in Yukon Standard Time.
  • Registry-linked details now offer both a human-facing registry page link and the raw API endpoint.

What This Means

The project is no longer at the “should we integrate the API?” stage. A working hybrid has been built. It keeps the stable geometry of the project map archive while layering in newer process state from the registry where a join is possible.

  • The map can now answer more than “where is the project?”
  • It can also answer “what stage is it in?” and “where do I go next?”
  • The QA outputs make it possible to improve source overlap methodically rather than by guesswork.

Remaining Gaps / Next Moves

  • Backfill or refresh additional API buckets if older-year completeness matters beyond the current cache coverage.
  • Investigate unmatched API projects to separate true no-geometry cases from join-field normalization problems.
  • Consider a registry-first “current activity” view for recent projects that may not yet exist in the shapefile archive.
  • Optionally surface stage/status filters directly in the UI now that stage metadata is already present.
